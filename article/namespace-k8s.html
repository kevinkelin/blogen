Use dns to access services in kubernetes

Publish: 2020-02-11 Classify at: k8s
Words:964 Read:≈ 5min

In the previous article Communication between pods in kubernetes , we use environment variables to resolve the IP of the service, but you can use environment variables such as A limitation, all pods must be in one namespace, that is to say, pods in the same namespace will share environment variables. How to access them if they are not in the same namespace? Let’s take a python flask application as an example. This time we put redis in the default namespace, and the flask application in the namespace of yyxtest.

#-*- coding:utf-8 -*-
#author:Yang
# datetime:2020/2/10 16:07
# software: PyCharm

from flask import Flask
from flask_redis import FlaskRedis
import time
import os

if os.environ.get("envname") == "k8s": # The description is in k8s
    redis_server = os.environ.get("REDIS_MASTER_SR_SERVICE_HOST")
    redis_port = os.environ.get("REDIS_MASTER_SR_SERVICE_PORT")
    REDIS_URL = "redis://{}:{}/{}".format(redis_server, redis_port, 1)
else:
    REDIS_URL = "redis://{}:{}/{}".format('127.0.0.1', 6380, 1)


app = Flask(__name__)
app.config['REDIS_URL'] = REDIS_URL
redis_client = FlaskRedis(app)

@app.route("/")
def index_handle():
    redis_client.set("reidstest",time.strftime("%Y-%m-%d %H:%M:%S",time.localtime(time.time())))
    name = redis_client.get("reidstest").decode()
    return "hello %s"% name

app.run(host='0.0.0.0', port=6000, debug=True)

The above is the previous code. Redis_server and redis_port are obtained by means of environment variables. The pod and service of the application that created flask have not specified the namespace. If they are not specified, they are created in the default namespace by default, because redis does not have either. Specify, so they can resolve the service address by sharing environment variables. Now we will create the python application in the namespace of yyxtest and see how it goes.

Package the previous python application docker build -t flaskk8s:dns .

This time, namespace: yyxtest is added to the metadata in deployment and service, they will be created in the namespace of yyxtest

Use kubectl get pods --namespace=yyxtest to view pods. At this time, pods display error directly. By viewing the logs in pods, it is found that redis_port = os.environ.get("REDIS_MASTER_SR_SERVICE_PORT") This line of code is not obtained To the value of REDIS_MASTER_SR_SERVICE_PORT, the returned value is None, so an error will be reported during the subsequent redis initialization, which also means that there is no REDIS_MASTER_SR_SERVICE_PORT environment variable in the pod in the yyxtesxt namespace.

We also create flask’s pod and service in the default namespace, which can be accessed normally at this time.

Use dns to resolve service addresses

In addition to using environment variables to resolve service addresses, DNS should be used more for resolution. When creating a redis service, Kubernetes will create a corresponding DNS entry in the form of <service-name> .<namespace-name>.svc.cluster.local, which means that if the container only uses <service-name>, it will be resolved to the service in the local namespace. For example, set it in the yaml file

1
2

metadata:
  name: redis-master-sr

A record of redis-master-sr.default.svc.cluster.local will be created, let’s try it with the pod in the default namespace

1
2

# ping redis-master-sr
PING redis-master-sr.default.svc.cluster.local (10.97.140.58) 56(84) bytes of data.

Try again with pods in the yyxtest namespace

1
2

# ping redis-master-sr
ping: redis-master-sr: Name or service not known

It means that the service name can only have dns records in the space where it is located (default in this example), and does not exist in its space (yyxtest in this example), but we note that in the default space redis-master -sr resolves to redis-master-sr.default.svc.cluster.local, so can the name redis-master-sr.default.svc.cluster.local be resolved normally in the non-default space?

Execute in the pod of yyxtest

1
2

ping redis-master-sr.default.svc.cluster.local
PING redis-master-sr.default.svc.cluster.local (10.97.140.58) 56(84) bytes of data.

It can also be parsed normally, so let’s modify the python code at this time

1
2
3
4

if os.environ.get("envname") == "k8s": # The description is in k8s
    REDIS_URL = "redis://{}:{}/{}".format("redis-master-sr.default.svc.cluster.local", 6379, 1)
else:
    REDIS_URL = "redis://{}:{}/{}".format('127.0.0.1', 6380, 1)

At this point, we can create applications in different namespaces from redis.
