Interface usage in golang

Publish: 2021-10-23 Classify at: Go
Words:1309 Read:≈ 7min

The interface in golang is very different from the interface in other object-oriented.

Definition of an interface

1
2
3
4

//Define an interface, it has a run method
type Runable interface {
	run()
}

Define a Runable interface, which has a method, run() , which has no parameters and no return value

A structure can define a method. If a structure defines the run() method, it means that the structure implements the Runable interface, which is not like a class in java or python, which is displayed when it is defined. Indicates which interface is inherited from.

Interfaces can have values

This is different from most object-oriented languages, their interfaces cannot be instantiated, but interfaces in Go can be “instanced”

1
2
3
4

func main() {
	var r Runable
	fmt.Println(r)
}

The print is nil, that is to say, the instantiated interface gets a null pointer, and the null pointer cannot directly call the interface method at this time, such as r.run(),

Later, if you want to assign the pointer, you also need to point to a structure variable that implements the run() method.

If an instance is pointed to, the run method can be called.

This means that the Cat class does not implement the run() method of the Runable interface

Implement polymorphism through interfaces

Polymorphism is very important in object-oriented languages. Because of polymorphism, the advantages of object-oriented can be brought into play. When we define a function, the parameters can use interface types. When the function is actually called, we only need to pass in an implementation. The structure of the interface is enough.

package main

import "fmt"

//Define an interface, it has a run method
type Runable interface {
	run()
}

type Person struct {
	name string
	age int
}

type Dog struct {
	name string
}

func (self *Person) run() {
	fmt.Println("Person run....")
}

func (self Dog) run() {
	fmt.Println("Dog run....")
}

// Define a test function, the parameter is the Runable interface type
func test(obj Runable) {
	obj.run()
}

func main() {
	p := Person{"yyx", 18}
	d := Dog{"wangwang"}
// Use the structure that implements the Runable interface as a parameter, when calling the corresponding method
	&test
	test(d)

}

The output of the above code is

Person run....
Dog run....

A test(obj Runable) function is defined in the above code, and the parameter is the Runable interface. In subsequent calls, you only need to pass a structure that implements the Runable interface. The run() method is called in the test function, and the The corresponding run method of the structure will be called.

func main() {
	var r Runable
	fmt.Println(r)
	p := Person{"yyx", 18}
	d := Dog{"wangwang"}
	r = d
	switch t:= r.(type) {
	case *Person:
		fmt.Printf("Type Person %T value %v \n", t,t)
	case Dog:
		fmt.Printf("Type Dog %T value %v \n", t,t)
	case nil:
		fmt.Println("t is nil....")
	default:
		fmt.Println("default case...")
	}
}

All structures except nil and default in case need to implement interface methods, otherwise a compilation error will occur.

func main() {
	var r Runable
	fmt.Println(r)
	p := Person{"yyx", 18}
	d := Dog{"wangwang"}
	// Determine if a value implements the Runable interface
	r = &p
	if _, ok := r.(Runable); ok{
		fmt.Println("p implements the Runable interface")
	}else{
		fmt.Println("p does not implement Runable interface")
	}
}

In my understanding, checking whether an interface value implements an interface here is equivalent to judging whether the interface is nil. When the interface value wants to point to a variable, if the variable does not implement the interface method, it is impossible. Point to success.

In the above code, if r = &p is commented out, then OK is false. At this time, it can only mean that nil has just been initialized, pointing to nil.
