Middleware development using gin to build api backend system

Publish: 2022-01-08 Classify at: Go
Words:840 Read:≈ 4min

In the previous articles, the default gin.Default() was basically used to initialize an engine, and then the Run method of the engine was called to run a web service. Now if there is a demand, some APIs only manage Only administrators can access, and non-administrators are very 401. We can write a method and call this method on the interface that requires authorization to verify the authority, but in gin, we can also use a more elegant method, middleware.

How does gin.Default() work?

Let’s first look at how gin.Default() works

	debugPrintWARNINGDefault()
	engine := New()
	engine.Use(Logger(), Recovery())

It first uses the New() method to initialize an engine, and then calls the engine’s Use method, loading two middleware,

func (engine *Engine) Use(middleware ... HandlerFunc) IRoutes {
	engine.RouterGroup.Use(middleware...)
	engine.rebuild404Handlers()
	engine.rebuild405Handlers()
	return engine
}

Loaded middleware in two gins, Logger() and Recovery()

The source code of Logger() is

1
2
3

func Logger() HandlerFunc {
	return LoggerWithConfig(LoggerConfig{})
}

The source code of Recovery() is

1
2
3

func Recovery() HandlerFunc {
	return RecoveryWithWriter(DefaultErrorWriter)
}

What is middleware

Middleware is a type of function, HandlerFunc , defined as type HandlerFunc func(*Context)

That is, the parameter of the function is Context, that is, if we want to customize a middleware function, we only need to return a function whose parameter is (*gin.Content).

What is middleware? When a request arrives at a route of the gin service, gin will process it according to the processing class defined in the route, taking the GET method as an example

1
2
3

func (group *RouterGroup) GET(relativePath string, handlers ... HandlerFunc) IRoutes {
return group.handle(http.MethodGet, relativePath, handlers)
}

handlers is an indeterminate parameter, which means that there can be multiple processing classes, and the processing class type here is also HandlerFunc, which is the same as middleware.

It can be understood that the middleware is multiple Handlers that process requests. A request goes through one Handler first, then the second and third, and finally returns the result to the caller.

The following two methods in middleware functions are more important

Next() method, after the middleware is processed, it will be processed by the next middleware

Abort() method, after calling this method, the subsequent HandlerFunc will not be processed

Two ways to define middleware

There are two common ways of middleware

  1. Define a function that returns gin.HandlerFunc,
  2. Directly define a function whose parameter is *gin.Context

package midwares

import (
	"fmt"
	"github.com/gin-gonic/gin"
	"time"
)

//Print time-consuming middleware

func Checktime() gin.HandlerFunc {
	//Here can handle some other logic
	return func(c *gin.Context) {
		start := time.Now()
		c.Next()
		spend:= time.Since(start).Microseconds()
		fmt.Printf("use: %d \n", spend)
	}
}

func Abort(c *gin.Context){
	fmt.Println("Abort middleware is used, no post-processing will be performed")
	c.JSON(200, gin.H{"msg": "no next"})
	c.Abort()
}

The effect of these two methods is the same, but the way to initialize and load gin is different

1
2
3

r := gin.Default()
r.Use(midwares.Checktime()) //Need to add ()
r.Use(midwares.Abort) // no need to add ()

Personally, I prefer the way of adding ().

Several ways of using middleware

There are two ways to use middleware, one is global middleware and the other is local middleware

For example, the way of using r.Use(midware) above is global. In this way, each route will follow the logic of the middleware.

Local middleware is used in route definition, and this definition is only valid on that route.

1

usv1.GET("/abort", midwares.Abort, usrv1_h.Abort)

I have defined two middleware above, Checktime is to record the time taken by the request, Abort middleware is to test the function of c.Abort, the route using this middleware will only go here, not It will go through the next route.

Access the above routing address /abort below, the console will output

The first line of output is the output of the Abort middleware, and the second line of output is the output of the Checktime middleware.

The return of http is the return defined by the Abort middleware, and the subsequent HandlerFunc is no longer executed.

Middleware-based authorization management

Create a middleware, if the verification is successful, continue to go down, if not the administrator, then don’t go down, return 401

func CheckAdmin() gin.HandlerFunc{
	return func(c *gin.Context) {
		username := c.Query("user")
		// Here can be judged from cookie or session
		if username == "admin"{
			c.Next()
		}else{
			c.JSON(401, gin.H{"msg": "No Private"})
			c.Abort()
		}
	}
}

Middleware execution order

When gin is initialized, it loads some middleware, defines the global middleware, and defines the routing middleware. So what is the execution order of these middleware?

The order of execution is Global Middleware > Routing Group Middleware > Routing Middleware.
