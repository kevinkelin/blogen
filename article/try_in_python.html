exception handling in python

Publish: 2017-08-19 Classify at: Python
Words:1704 Read:≈ 8min

Writing programs must be exposed to exception handling. Fortunately, exception handling in python is relatively simple, but some details still need to be looked at. Here is a record of the use of try in python.

The simplest, try…except

Take a look at the execution of the following code

1

def test():
    print 'yangyanxing'
    print 5/0
    print 'end script....'

if __name__ == '__main__':
    test()

When the program runs to print ‘yangyanxing’, there is no error, and the output can be normal But when it reaches 5/0, it will trigger an error of division by 0. We all know that 0 cannot be used as a divisor, so the program will crash here. The following print 'end script... ' will not be executed Executing the program gives the following output

yangyanxing
Traceback (most recent call last):
  File "try.py", line 11, in <module>
    test()
  File "try.py", line 7, in test
    print 5/0
ZeroDivisionError: integer division or modulo by zero

How to make the program run normally? It’s very simple, add a try to capture the places where exceptions may occur

def test():
    print 'yangyanxing'
    try:
        print 5/0
    except Exception as e:
        print e
    
    print 'end script....'

if __name__ == '__main__':
    test()

will get the following output

yangyanxing
integer division or modulo by zero
end script....

In this way, when the program runs to the line of code with the error, the code defined at except will be executed, that is, print e, and then the program will execute print 'end script....'

However, you can’t add a separate try statement to each step of the operation, so the program will look very strange, and sometimes you don’t know what will go wrong, so the scope of try can be increased.

Please see the trackback information below 1111, which line is the error marked here, what is the error, be sure to make good use of trackback

try…except…finally

This one has one more finally than the previous try…except, and it is more powerful in function. This finally is not necessary, and it is also determined according to the actual situation when using it. And there are a lot of things to pay attention to after adding finally. Finally means that no matter whether the program has an exception or not, its code must be executed at the end.

see example like this

#coding:utf-8

import traceback

def test():
    try:
        print 'yangyanxing'
        print 5/0
        print 'hello world'
    except Exception as e:
        print traceback.format_exc()
    finally:
        print 'end script....'

if __name__ == '__main__':
    test()

The output obtained is

yangyanxing
Traceback (most recent call last):
  File "/Users/yangyanxing/Desktop/try.py", line 8, in test
    print 5/0
ZeroDivisionError: integer division or modulo by zero

end script....

Note that print 'end script....' is executed here, and the program is interrupted when the error occurs when printing 5/0, and the except code is executed. After the except code is executed, the finally code will be executed. , and the code behind the exception will not be executed

#coding:utf-8

import traceback

def test():
    try:
        if 2>1:
            return 1
        else:
            return 0

    except Exception as e:
        print traceback.format_exc()
        return 2
    finally:
        print 'end script....'
        return 3

if __name__ == '__main__':
    print test()

What is its return value? 0?1?2?3?

Take a look at the result of the program running

end script....
3

The result is 3

That is to say, even if the return value is defined in try, it is useless. The program finally returns the return value executed in finally, so if finally is used, it is best not to define the return value in try and except. Save it to a variable and return in finally.

#coding:utf-8

import traceback

def test():
    print 'start test...'
    test2()
    print 'end script...'

def test2():
    try:
        print 'start test2...'
        print 5/0
        print 'end test2...'
    except Exception as e:
        print traceback.format_exc()

if __name__ == '__main__':
    test()

What will it turn out to be?

start test...
start test2...
Traceback (most recent call last):
  File "/Users/yangyanxing/Desktop/try.py", line 13, in test2
    print 5/0
ZeroDivisionError: integer division or modulo by zero

end script...

Note that in the test2() function, the division by 0 error has been caught and processed, so in the test function, it will not be affected, and the end script can also run normally. However, for some complex programs, when an exception occurs in the test2 function, the function that calls it will execute normally only when test2 runs normally in the next operation, which is embarrassing. How to deal with it?

The answer is to use raise in test2 to throw this exception to let the caller know that it went wrong.

#coding:utf-8

import traceback

def test():
    try:
        print 'start test...'
        test2()
        print 'end script...'
    except Exception as e:
        print traceback.format_exc()

def test2():
    try:
        print 'start test2...'
        print 5/0
        print 'end test2...'
    except Exception as e:
        print traceback.format_exc()
        raise e

if __name__ == '__main__':
    test()

The output obtained is

start test...
start test2...
Traceback (most recent call last):
  File "/Users/yangyanxing/Desktop/try.py", line 16, in test2
    print 5/0
ZeroDivisionError: integer division or modulo by zero

Traceback (most recent call last):
  File "/Users/yangyanxing/Desktop/try.py", line 8, in test
    test2()
  File "/Users/yangyanxing/Desktop/try.py", line 20, in test2
    raise e
ZeroDivisionError: integer division or modulo by zero

The above output is printed in test2(), and the following is printed by the test() function. Here, the print 'end script...' in the test() function is not executed because an exception is caught.

Here, raise is also used flexibly according to the business process. Sometimes an exception will be thrown when the large process fails. For example, in a recent project, the virtual machine starts and executes the corresponding test code. Encountered that the virtual machine did not start successfully, but the program will still execute the following code. The reason for the check is that the virtual machine will roll. It caught the exception, but did not throw it, and then the upper layer did not judge it when it was called. Whether the roll is successful or not, this will cause the program to be executed incorrectly.

custom exception

Sometimes when these exceptions defined by the system cannot meet the requirements, you need to define some exceptions yourself, which are used more often. For example, the following scenario should be designed. When the loop reaches 5 times, an exception of ‘I quit my job, I am too tired’ is thrown.

#coding:utf-8

import traceback
import time
def test():
    try:
        print 'start test...'
        test2()
        print 'end script...'
    except Exception as e:
        print traceback.format_exc()

def test2():
    try:
        times = 10
        while times > 0:
            print times
            if times == 5:
                raise TypeError('i don\'t want to work....')
            times -=1
            time.sleep(1)
    except Exception as e:
        print traceback.format_exc()
        raise e

if __name__ == '__main__':
    test()

#coding:utf-8

import traceback
import time

class IdontWantWork(Exception):
    def __init__(self,err=''):
        Exception.__init__(self,err)


def test2():
    try:
        times = 10
        while times > 0:
            print times
            if times == 5:
                raise IdontWantWork('I quit, I'm too tired')
            times -=1
            time.sleep(1)
    except Exception as e:
        print traceback.format_exc()

if __name__ == '__main__':
    test2()
