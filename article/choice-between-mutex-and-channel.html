The choice of lock Mutex and channel channel in golang

Publish: 2021-10-23 Classify at: Go
Words:562 Read:≈ 3min

Both Mutex and channel can protect resources in a concurrent environment and avoid race conditions. Channel has always been sought after in golang, but since both can solve the problem, why do you need two things?

Looking at some articles, I found that sometimes channels are too sought after. Sometimes you should use Mutex or Mutex, don’t use channel just to use channel, you need to distinguish between different scenarios

Let’s take an example of decrementing a common variable by one operation under multiple coroutines

package main

import (
	"fmt"
	"sync"
)

var total int = 1000
var wg sync.WaitGroup

func main() {
	for i:=0; i<1000; i++{
		wg.Add(1)
		go func() {
			defer wg.Done()
			total -= 1
			}()
	}
	wg.Wait()
	// print the value of total
	fmt.Println(total)
}

There is a public variable total of 1000, then start 1000 coroutines, each coroutine decrements the total, we want to get a result of 0, but since multiple coroutines operate a variable at the same time, in the case of no lock, The final result is not 0 in most cases, and the result is different each time.

Using lock Mutex

Since there is a situation in which the same resource is operated, it is necessary to add a lock

package main

import (
"fmt"
"sync"
)

var total int = 1000
var wg sync.WaitGroup
var lock = sync.Mutex{}

func main() {
	for i:=0; i<1000; i++{
		wg.Add(1)
		go func() {
			lock.Lock()
			defer wg.Done()
			defer lock.Unlock()
			total -= 1
		}()
	}
	wg.Wait()
	// print the value of total
	fmt.Println(total)
}

In the anonymous function go func, lock.Lock() is first locked, and then the unlocking is delayed, and the Done operation of wg is performed.

This time, because of the lock, the final output is 0.

func main() {
	var done = make(chan bool, 1)
	done <- true
	for i:=0; i<1000; i++{
		wg.Add(1)
		go func() {
			<- done //Under multi-coroutines, it will block if it can't be retrieved
			defer wg.Done()
			total -= 1
			done <- true
		}()
	}
	wg.Wait()
		// print the value of total
	fmt.Println(total)
}

Using channels in anonymous functions to protect shared data

Mutex and channel selection

  1. Channels are an advanced concept in Go. Some programs in Go only use Mutex. Go’s Channels are attractive because they provide built-in thread safety and encourage single-threaded access to shared critical resources. But compared to Mutex, Channel causes performance degradation. Using Mutex is useful when only a few shared resources need to be locked. Feel free to use sync.Mutex if Mutex suits your needs.
  2. If you need to flow data resources between coroutines, you need to use channels
  3. Channel pays attention to data flow. If there is data flow in the task processing model, use channel to solve it;
  4. The control of the data needs to be passed in multiple gorulines and processed by channel;
