Detailed explanation of the use of asyncio in python

Publish: 2019-06-08 Classify at: Python
Words:1507 Read:≈ 8min

In the asynchronous practice of python, every time you look at asynicio, you will have new gains. This article summarizes the use of this library recently.

Some core concepts

Definition of async function

The definition of ordinary functions is to use the def keyword, asynchronous functions, coroutine function (Coroutine) is essentially a function, the characteristic is that the execution right can be handed over to other coroutines in the code block, use async def to definition

# Ordinary function definition
def add2(x):
    print(x+2)
    return x+2

# Definition of async function
async def add3(x):
    print("in async fun add")
    return x+3

How to call a coroutine and get its result? Calling an ordinary function only needs result = add2(2), then the function can be run, and return the result 4 to result, if you use result = add3(2), then print result at this time ? What I get is a coroutine object, <coroutine object add3 at 0x000002ED564A5048>, which is not the result of 2+3=5. How can I get the result? The coroutine function needs to be executed in the event loop to execute.

Event loop Eventloop

Eventloop is the core of the asyncio application. Register some asynchronous functions to this event loop, and the event loop will execute these functions in a loop. When a function is executed, if it is waiting for I/O to return, such as it is making a network request, Or sleep operation, the event loop will suspend its execution to execute other functions; when a function completes I/O, it will resume and continue to execute the next time it loops. Therefore, these asynchronous functions can run cooperatively: this is the goal of the event loop.

Return to the above function, to get the function execution result, you need an Eventloop

# coding:utf-8

import asyncio
import time


async def testa(x):
    print("in test a")
    await asyncio.sleep(3)
    print("Resuming a")
    return x


async def testb(x):
    print("in test b")
    await asyncio.sleep(1)
    print('Resuming b')
    return x


async def main():
    start = time.time()
    resulta = await testa(1)
    resultb = await testb(2)
    print("test a result is %d"%resulta)
    print("test b result is %d"%resultb)
    print("use %s time"%(time.time()-start))

if __name__ == '__main__':
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

This code defines two coroutines and puts them in another coroutine main function. To get the result of their running, the event loop is characterized by waiting when it encounters a certain I/O (such as here the asyncio.sleep() function), you can execute other functions, so that the time required for the entire function to execute should be the one with the longest execution time among all the coroutines. For the above code, a sleep 3 seconds, a sleep for 1 second, the total time should be a little more than 3 seconds, but is this the result?

It takes a little more than 4 seconds, and the testa function is executed first, and then the testb function is executed. It is executed serially and sequentially, and it is not executed concurrently as we imagined. How should it be executed concurrently?

async def main():

    resulta,resultb = await asyncio.gather(testa(1),testb(2))

if __name__ == '__main__':

It can be seen that testa and testb are running synchronously. Since testb only sleeps for 1 second, testb first outputs Resuming b, and finally returns the result of each coroutine function. Note that here is the gather() function After each coroutine function in it is executed, it will result in a list. The order of the values ​​in the list is the same as the order of the coroutines placed in the gather function.

async def main():
    start = time.time()

    taska = asyncio.ensure_future(testa(1))
    taskb = asyncio.ensure_future(testb(2))

    print(taska)
    print(taskb)
    print(taska.done(), taskb.done())
    await taskb
    await taska
    print(taska.done(), taskb.done())

    print(taskb.result())
    print(taska.result())
    print("use %s time" % (time.time() - start))

if __name__ == '__main__':
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

Use asyncio.ensure_future(testa(1)) to return a task object. At this time, the task enters the pending state and is not executed. At this time, print(taska) gets <Task pending coro=<testa() running at F :/python/python3Test/asynctest.py:7>> For some time, taska.done() returns False, indicating that it is not over yet. When await taska is called, it means that the coroutine will be executed. After the execution ends, taska.done() returns True, then you can call taska.result() to get the return value of the function. If the result() method is called before the coroutine ends, an exception will be thrown, raise InvalidStateError('Result is not ready .').

What is the difference between using wait and gather? First of all, gather requires all tasks to be executed. If a coroutine function crashes, an exception will be thrown and there will be no result. wait can define the timing of the function’s return, which can be FIRST_COMPLETED (the first to end), FIRST_EXCEPTION (the first to be abnormal), ALL_COMPLETED (all executed, the default)

# coding:utf-8

import asyncio
import time


async def testa(x):
    print("in test a")

    await asyncio.sleep(3)
    print(1/0)
    print("Resuming a")
    return x


async def testb(x):
    print("in test b")
    await asyncio.sleep(1)
    print(1/0)
    print('Resuming b')
    return x


async def main():
    start = time.time()
    done, pending = await asyncio.wait([testa(1),testb(2)],return_when=asyncio.tasks.FIRST_EXCEPTION)
    print(list(done))
    print(list(pending))
    print("use %s time" % (time.time() - start))

if __name__ == '__main__':
    loop = asyncio.get_event_loop()
    loop.run_until_complete(main())

This code requires the result when the first exception occurs, the function as a whole will not crash, but it is an exception object if you want to get the result here.
