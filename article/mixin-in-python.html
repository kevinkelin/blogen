Usage of Mixin class in python

Publish: 2022-07-10 Classify at: python
Words:1975 Read:≈ 10min

Mixin is nothing new in python, it is just a manifestation of programming ideas, using its multi-inheritance language features to achieve a flexible programming method.

Recently, I was looking at the source code of sanic and found that there are many Mixin classes, which probably look like this

So I researched this kind of Mixin, but there is actually nothing new. Mixin, also known as Mixin, is just a reflection of programming ideas, but there are still some points to pay attention to in the process of use.

As we all know, python is a language that allows multiple inheritance. A class can inherit multiple classes, which is different from java. A java class can only have one parent class, but java has the concept of interface, and a class can implement There are multiple interfaces, but the java interface is only the signature of the defined function, and there is no specific implementation. The specific implementation needs the corresponding class to complete.

Python is different. A class can have multiple parent classes, and mixing classes is a programming pattern that only allows multiple inheritance languages.

In order to better understand mixing, let’s take an example from life - a mobile phone. A mobile phone has many functions and is composed of different hardware. A mobile phone requires splicing of various hardware.

If we abstract these functions into classes, then we can have the following way of writing. For simplicity, we only list the functions of making and receiving calls and sending and receiving text messages.

class Tel:
    def telfunc(self):
        print("I can make calls")
        
class SMS:
    def smsfunc(self):
        print("I can send text messages")
        
class Phone:
    def __init__(self, sn):
        self.sn = sn

In the above code, there are three classes, Tel class, which has a telfunc method to indicate the ability (or function) to make and receive calls, and smsfunc for SMS class to indicate that the SMS class has the ability to send text messages. The Phone class is a mobile phone class, it should have the ability to make calls and send text messages, but if we define the Phone class in the above way, this class does not have the ability to make calls and send and receive text messages.

What can we do to make the Phone class have the ability to make calls and send text messages?

We can redefine two methods telfunc and smsfunc in the Phone class, that is, write the methods in the Tel class and SMS class again, which is actually not in line with the concept of don't repeate youself .

Under normal circumstances, we let the Phone class inherit the Tel class and the SMS class, so that the Phone class automatically has the ability to make calls and send text messages.

class Tel:
    def telfunc(self):
        print("I can make calls")


class SMS:
    def smsfunc(self):
        print("I can send text messages")


class Phone(Tel, SMS):
    def __init__(self, sn):
        self.sn = sn

    def welcome(self):
        print("welcome {}".format(self.sn))


p = Phone("xiaomi")
p.telfunc()
p.smsfunc()
p.welcome()

'''
i can make calls
I can send text messages
welcome xiaomi
'''

The definition of such a class is what we call mix-in. The call function and SMS function are added to the mobile phone, so that the mobile phone has the function of making and receiving calls and sending text messages. This mixed-in coding idea can sometimes reduce a lot of code. It’s very convenient to “mix-in” a class into a class based on what functionality it needs.

Usually, we will name the mixin class ending with Mixin, like the above code we will write the following

class Phone(TelMixin, SMSMixin):
    def __init__(self, sn):
        self.sn = sn

    def welcome(self):
        print("welcome {}".format(self.sn))

The class ending with Mixin is generally the kind of function that is relatively single, and generally has a certain type of function, such as the sanic mixed class introduced at the beginning

RouteMixin is a functional class related to routing, MiddlewareMixin is a class related to middleware, and ListenerMixin is a class related to listeners. The methods in these classes focus on their own related functions. If any class needs these functions, it is defined when Inherit from these classes.

Some people will ask, I don’t want to inherit more, it’s too troublesome to manage MRO, I just want to inherit alone, I define a unified parent class, it has the functions of making and receiving calls, sending and receiving text messages, listening to songs, and then letting Isn’t it bad for the mobile phone to inherit this class? If I want to create an ipod class, the ipod class There is no function of receiving and sending text messages, only the function of listening to songs, so how should I define this class at this time?

Is it necessary to define an ipod class, and then write its listening method songfunc? If we use the mixed programming idea at this time, then we can completely define the ipod class without adding to make calls and send and receive text messages. class will do.

If there are other classes, such as walkie-talkies, it only has the function of making and receiving calls, then we only need to mix the function of making and receiving calls into the walkie-talkie class, class Intercom(TelMixin):

But some people will ask again, such a mix-in, although some code is written less, but what if the corresponding implementation of the subclass is not completely consistent with the parent class? For example, although the walkie-talkie can talk, it is a one-way call, not a two-way call.

At this time, for the situation that the subclass implementation is inconsistent with the parent class, then the subclass needs to rewrite the parent class method, that is, inheritance and polymorphism in OOP. Strictly speaking, python does not have polymorphism, or it is naturally is polymorphism.

class TelMixin:
    def telfunc(self):
        print("I can make calls")


class SMSMixin:
    def smsfunc(self):
        print("I can send text messages")


class SongMixin:
    def songfunc(self):
        print("I can play music")


class Intercom(TelMixin):
    def __init__(self, sn):
        self.sn = sn

    def telfunc(self):
        print("Intercom{} call".format(self.sn))


class Phone(TelMixin, SMSMixin, SongMixin):
    def __init__(self, sn):
        self.sn = sn


class Ipod(SongMixin):
    def __init__(self, sn):
        self.sn = sn


def testTelfunc(o):
    o.telfunc()


d = Intercom("moto")
p = Phone("huawei")
ipod = Ipod("apple")
testTelfunc(d)
testTelfunc(p)
testTelfunc(ipod)

In the above code, we rewrote the telfunc method of the walkie-talkie class, but did not implement the call function for the Ipod class, and then wrote a testTelfunc(o) method to call the call function of the incoming parameters, and get the following output

When the Intercom object of the walkie-talkie is passed in to testTelfunc, since this class rewrites the telfunc method, the telfunc method of the subclass is called here. When the Phone class object is passed in, because this class inherits the TelMixin class , and the telfunc method is not rewritten, so the telfunc method in the TelMixin class will be called here, but when the Ipod class is passed in, this class neither inherits TelMixin nor has its own telfunc method, so it crashes. This is also an alternative embodiment of polymorphism, but it is not as strict as in java.

Difference between Mixin and Inheritance

Having said so many mixins, in fact, it is essentially inheritance, but this kind of inheritance exists in programming languages ​​that allow multiple inheritance. If there is a difference, there is essentially no difference. If you insist on saying some difference, there is actually a little bit.

  1. First of all, it is best not to have an initialization method in the Mixin class, that is, the __init__ method, because if the Mixin class defines an initialization method, then it is best to call the parent class in the subclass, which is the initial method of the mixed class super ().__init__(), of course it doesn’t matter if you don’t call it, the IDE will only give you a hint, and there is no syntax error.
  2. It is best not to instantiate the Mixin class separately, because the original design of the mixin class is to let subclasses inherit, and then the subclass can enrich the functions of the mixin class, and the mixin class is only for the subclass to have certain functions. of.
  3. The Mixin class is preferably a collection of a certain type of functions, such as the RouteMixin mixin class in the sanic source code above, which only defines functional methods related to routing. The advantage of this is that there will be no methods with the same name. If it is mixed into the class If there is a method with the same name, then you will cause trouble for yourself, and it is not easy to troubleshoot the problem. In fact, this is also more convenient than inheriting from a single class.

Having said so much about the benefits of Mixin, is there any downside to it? In fact, in my opinion, there is a problem, and it’s not bad, it’s just a problem of habit. I used to write more code with single inheritance, and I’m not used to this method of multiple inheritance, but if everything follows the corresponding rules Operation is not a big problem.

Another problem is the splitting of functions, which is a test of the programmer’s overall grasp of the code at the beginning of the design. Just like a single application that wants to be split into microservices, the boundary problem may be the most troublesome at the beginning. Well, I will write a lot of classes, let’s break it down, and I will return to the single inheritance thinking mode, so this is also a question of experience.

Summarize

Mixin can also be said to be a programming mode, and it is not a new syntax. Using mixin classes well can make your code structure clear and functions clear. Therefore, when designing classes in the future, you should consider the implementation of Mixin classes.
