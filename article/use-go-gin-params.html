Use gin to build api background system to get request parameters

Publish: 2021-12-16 Classify at: Go
Words:975 Read:≈ 5min

The previous article recorded the construction of the gin framework and the basic use of routing. This article mainly records the problem of parameter transmission in the routing process, including GET and POST parameters, and the use of path parameters.

Get GET parameters

The parameters of the get request are written in the url, such as http://xxxx.com/path?id=1234&name=Manu&value=*111

The gin framework mainly uses the following methods to obtain get parameters:

  • func (c *Context) Query(key string) string
  • func (c *Context) DefaultQuery(key, defaultValue string) string
  • func (c *Context) GetQuery(key string) (string, bool)

Write a handler class:

1
2
3
4

func (user) Get(c *gin.Context) {
	name := c.Query("name")
	c.JSON(200, gin.H{"msg": "Hello User:"+ name})
}

When the route is accessed normally, such as /name=yyx, it will return msg: "Hello User: yyx", if the name parameter is not added, the name is an empty string and return msg: "Hello User:"

Sometimes we need to have a default parameter when no parameters are passed, then the DefaultQuery method will be used at this time.

1
2
3
4

func (user) Get(c *gin.Context) {
	name := c.DefaultQuery("name", "yyx")
	c.JSON(200, gin.H{"msg": "Hello User:"+ name})
}

At this time, if the name parameter is not passed, the default yyx will be used instead.

The bottom layer of Query and DefaultQuery is the GetQuery method called. The second return value of the GetQuery method indicates whether the parameter you want to get exists. With this parameter, we can do some judge

1
2
3
4
5
6
7

func (user) Get(c *gin.Context) {
	if name, ok := c.GetQuery("name"); ok{
		c.JSON(200, gin.H{"msg": "Hello User:"+ name})
	}else{
		c.JSON(200, gin.H{"msg": "name parameter is missing"})
	}
}

Get POST parameters

GET is relatively simple to obtain parameters, and the parameters passed by POST are more complex. The post parameters are mainly application/x-www-form-urlencoded and multipart/form-data, these two types can be set by setting Content-Type in the header.

Corresponding to the GET parameter, there are also three ways to obtain the post parameter.

  • func (c *Context) PostForm(key string) string
  • func (c *Context) DefaultPostForm(key, defaultValue string) string
  • func (c *Context) GetPostForm(key string) (string, bool)

In most cases, application/x-www-form-urlencoded is used, which can be received using the PostForm method

1
2
3
4

func (user) Post(c *gin.Context) {
	name := c.PostForm("name")
	c.JSON(200, gin.H{"msg": "hi,"+name})
}

If no name parameter is passed, a default value can be set

1
2
3
4

func (user) Post(c *gin.Context) {
	name := c.DefaultPostForm("name", "Yanxing Yang")
	c.JSON(200, gin.H{"msg": "hi,"+name})
}

You can also use GetPostForm to determine whether the parameter exists

1
2
3
4
5
6
7

func (user) Post(c *gin.Context) {
	if name,ok := c.GetPostForm("name"); ok{
		c.JSON(200, gin.H{"msg": "hi,"+name})
	}else{
		c.JSON(200, gin.H{"msg": "parameter missing"})
	}
}

Parameters are bound to the structure

The above methods of obtaining parameter values ​​in get or post are all assigned one by one. When there are many parameters, it will be very troublesome. At this time, you can define a structure and then bind the parameters to the structure.

Such as defining the student structure

1
2
3
4

type student struct {
	Name string `json:"name" form:"name"`
	Age int `json:"age" form:"age"`
}

Here, when defining the structure, tag is added, so that the parameters can be recognized in the form of json or form

1
2
3
4
5
6
7
8

func (user) Post(c *gin.Context) {
	s := student{}
	if err := c.ShouldBind(&s); err==nil{
		c.JSON(200, gin.H{"msg": "hi,"+s.Name})
	}else{
		c.JSON(200, gin.H{"msg": "parameter missing"})
	}
}

restful style urls

Restful style urls usually write resource information into the url, such as /user/:id . In this case, it is impossible to add a route for each id pair. In this case, we can use wildcard routing

Two steps are required, the first step is to set the routing information

1
2
3
4
5
6
7
8
9

func main() {
	r := gin.Default()
	userH:= handlers.NewUser()
	userG := r.Group("/user")
	userG.GET("/list", userH.Get)
	userG.POST("/post", userH.Post)
	userG.GET(":id/info", userH.Info) //Set wildcard routing here
	r.Run(":8080")
}

Here, set :id as the route parameter to be matched, and then get the id parameter in the corresponding processing class

1
2
3
4

func (user) Info(c *gin.Context) {
	id := c.Param("id")
	c.JSON(200, gin.H{"msg": "User id is "+id})
}

Of course, you can also set multiple parameters userG.GET(":name/:age/info", userH.Infos)

But this kind of routing parameter needs to pay attention to some problems, if the following route is defined

1
2

userG.GET(":name/:age/info", userH.Infos)
userG.GET(":id/infos", userH.Info)

Then the startup will fail, prompting panic: ':id' in new path '/user/:id/infos' conflicts with existing wildcard ':name' in existing prefix '/user/:name'

It matches according to the above path and finds that the name and id are in the same position, so it will prompt that the id parameter can no longer be defined, that is to say, the parameter of a certain position can only be defined once and cannot be defined repeatedly.

Second question, you can define a route as follows

userG.GET("abc:id/infos", userH.Info) Then in the path matching, you can match /abc10/infos, the id is 10, and the abc string is not added, it cannot be matched.

The above is a common method for recording some parameters on the gin framework, but there are still some problems. For example, in the parameter matching process, if the age parameter is required to be an int value and is between 0 and 100, it is not yet possible to verify it. The judgment can only be made in the corresponding processing class. The next article will record the use of the parameter validator.
