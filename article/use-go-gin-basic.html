Use gin to build the framework of the API background system

Publish: 2021-12-15 Classify at: Go
Words:1231 Read:≈ 6min

Gin is a popular framework in golang. Many systems are developed under this framework. This framework feels to me like Tornado’s position in python. It has basic functions, but if you want to use it well, I still need to develop a lot of my own functions and middleware. After reading a lot of tutorials, I want to record the learning process.

This series simply implements some usable methods for background services, such as obtaining parameters, database query, etc. The overall project is not about the structure, at most it is a quick start, and the directory structure is not so particular, because beginners, So there will definitely be a lot of questions. Later, I will practice a little bit of repair.

download gin

The first one, simpler, build the framework first and write a hello world.

The project is based on the go module method. Run go mod init gintest in an empty directory to initialize the project, and go will automatically generate a go.mod file

Add gin to go.mod file

module gintest

go 1.15

require (
github.com/gin-gonic/gin v1.7.4
)

Then use the go mod download command to download.

Or run the go get github.com/gin-gonic/gin@v1.7.4 command directly in the project directory, the gin package will also be downloaded and added to the go.mod file automatically.

Build http service

Create a new server.go file and write the following code

package main

import "github.com/gin-gonic/gin"

func main() {
    r := gin.Default()
    r.GET("/", func(c *gin.Context) {
        c.JSON(200, gin.H{
            "Msg":"Hello World",
        })
    })
    r.Run(":8080")
}

The code is very simple, running go run server.go will start an http service on port 8080, and using a browser to access http://127.0.0.1:8080/ will return json data

In the above code, gin.Default() is used to create a default gin application, which initializes a lot of settings at the bottom of the application, so I won’t explore it in depth here.

A route is registered here, /, its processing is handled by an anonymous function, and the GET method in gin is defined as

func (group *RouterGroup) GET(relativePath string, handlers ... HandlerFunc) IRoutes {
return group.handle(http.MethodGet, relativePath, handlers)

404 Configuration

The default 404 processing will print 404 not found. More often, we want to customize the processing of 404 pages, which can be achieved by setting the r.NoRoute method

1
2
3
4
5

r.NoRoute(func(c *gin.Context) {
  c.JSON(404, gin.H{
    "Msg": "The page not found",
  })
})

In this case, custom content will be displayed when accessing a non-existing route.

routing group

In many cases, we have such a requirement that the route to be accessed has a uniform prefix, such as /api/v1, or /user, of course, you can write the prefix uniformly when defining the route, but this kind of way, if the prefix is ​​modified later, it will be changed in many places. The concept of routing groups can be used here.

func main() {
r := gin.Default()
r.GET("/", func(c *gin.Context) {
	c.JSON(200, gin.H{
	"Msg": "Hello world",
	})
})
apiv1 := r.Group("/api/v1")
{
	apiv1.GET("/", func(c *gin.Context) {
		c.JSON(200, gin.H{"msg": "hello api v1"})
	})
}
r.Run(":8080")
}

The above code defines a routing group, /api/v1, and then defines the corresponding processing method under the routing group. Here, the url of http://127.0.0.1:8080/ returns "Msg ": "Hello world", /api/v1/ This route returns "msg": "hello api v1"

Handling class encapsulation

The above routing processing classes all use anonymous functions. If the processing logic is relatively simple, you can write it like this, but it is still not recommended to write an anonymous function for processing. Later, as the project grows larger, the routing definitions here will become more and more The more, if there are a large number of anonymous functions, it will look more messy.

Here we can encapsulate the processing handler.

First create a handlers folder and write the following code

package handlers

import "github.com/gin-gonic/gin"

type ApiV1 struct {

}

func (ApiV1) Get(c *gin.Context) {
	c.JSON(200, gin.H{"msg": "handlers get request!"})
}

A structure of ApiV1 is defined here, and then a Get method is defined, and then in the entry server.go file, the original code is modified,

func main() {
	r := gin.Default()
	apiv1_h := handlers.ApiV1{}
	r.GET("/", func(c *gin.Context) {
		c.JSON(200, gin.H{
			"Msg": "Hello world",
		})
	})
	apiv1 := r.Group("/api/v1")
	{
		apiv1.GET("/", apiv1_h.Get)
	}
	r.Run(":8080")
}

Use apiv1_h := handlers.ApiV1{} to initialize an ApiV1 structure object. When the route is defined later, you can directly use the Get method of the object. It is best to put the same prefix in a bracket {} , method view

{
  apiv1.GET("/", apiv1_h.Get)
}

Note here that the handler is apiv1_h.Get , not apiv1_h.Get(c, *gin.Context)

Start the service using http.Server

The above code starts the service in the way of r.run(":8080"), but sometimes we want to do some personalized settings, such as some read and write timeouts, of course, we can also call gin.New() The method can be set in the object obtained, or you can directly use the definition in http.Server.

func main() {
	r := gin.Default()
	apiv1_h := handlers.ApiV1{}
	apiv1 := r.Group("/api/v1")
	{
		apiv1.GET("/", apiv1_h.Get)
	}
	s := &http.Server{
		Addr: ":8080",
		Handler: r,
		ReadTimeout: 10*time.Second,
		WriteTimeout: 10*time.Second,
	}
	err := s.ListenAndServe()
	if err != nil {
		return
	}
}

web page jump

The above code returns json data when processing requests. Since the mainstream development uses the front-end and back-end separation mode, the html template method is not used here to return data.

When using a third-party login, it all involves web page jumping, and web page jumping is also very simple, modify the apiv1.go file

package handlers

import "github.com/gin-gonic/gin"

type ApiV1 struct {

}

func (*ApiV1) Get(c *gin.Context) {
	c.Redirect(301, "https://www.baidu.com")
}

In this way, when the corresponding route of the method is used, it will jump to Baidu.

It should be noted here that the status code must be processed between 300-308, otherwise an error will be reported

func (r Redirect) Render(w http.ResponseWriter) error {
	if (r.Code < http.StatusMultipleChoices || r.Code > http.StatusPermanentRedirect) && r.Code != http.StatusCreated {
		panic(fmt.Sprintf("Cannot redirect with status code %d", r.Code))
	}
	http.Redirect(w, r.Request, r.Location, r.Code)
	return nil
}
